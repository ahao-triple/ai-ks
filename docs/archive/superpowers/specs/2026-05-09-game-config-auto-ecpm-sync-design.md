# AI-KS 游戏配置中心与自动 ECPM 同步设计

## 背景

当前系统已经支持管理员手动刷新快手 ECPM，并记录同步任务历史。下一阶段把“游戏相关配置”集中到每个游戏自己的配置页，同时为每个游戏增加可配置的自动 ECPM 同步能力。

管理员期望：

- 游戏列表里每个游戏都有一个“配置”入口。
- 配置页集中管理该游戏需要的运营配置，并清楚说明每项配置的作用。
- 自动 ECPM 同步默认关闭，管理员开启后才执行。
- 每个游戏可以单独配置同步频率。
- 默认频率为 3 小时，频率从预设选项中选择。
- 自动同步失败后只记录失败，由管理员自行处理。

## 目标

- 新增独立的游戏配置视图，采用左侧导航、右侧配置内容的结构。
- 把游戏基础信息、预算与结算、ECPM 同步、审计/任务历史集中到该游戏配置页。
- 支持每个游戏独立启停自动 ECPM 同步。
- 支持每个游戏独立配置自动同步频率。
- 自动同步开启后立即执行一次最近一个频率窗口的同步。
- 手动 ECPM 刷新迁入游戏配置页，并支持选择最近多久的数据范围。
- 复用现有 `KuaishouEcpmSyncJob` 任务历史记录自动和手动同步结果。

## 非目标

- 不引入外部队列、外部 cron 或分布式锁。
- 不做失败任务自动重试。
- 不做自动失败补拉历史数据。
- 不开放任意自定义同步频率或任意日期时间范围。
- 不重构现有结算、预算、ECPM 入库规则。
- 不一次性实现复杂权限矩阵；本阶段仍以管理员权限为边界。

## 产品结构

### 游戏列表入口

运营台现有游戏列表中，每个游戏行增加“配置”按钮。点击后进入独立的游戏配置视图，保留当前 Dashboard 外壳，不跳出管理员工作台。

游戏列表仍用于浏览游戏和快速进入配置，不再承载所有游戏操作表单。长期方向是让游戏相关操作优先在单个游戏配置页内完成。

### 游戏配置视图

配置视图采用已确认的 **左侧导航 + 右侧配置内容** 布局。

左侧导航模块：

- 基础信息
- 预算与结算
- ECPM 同步
- 审计/任务历史

右侧内容只展示当前模块，避免单页过长。每个字段附近提供后台式说明文字，说明该配置影响什么业务行为。

### 基础信息

展示和维护：

- 游戏名称。
- `game_app_id`，只读展示。
- `game_secret`，可更新。
- 所属公司，只读展示。

说明：

- `game_app_id` 用于识别快手小游戏。
- `game_secret` 用于游戏端登录校验和相关服务端校验。
- 修改密钥会影响后续游戏端登录请求，旧客户端需要确认同步更新。

### 预算与结算

展示和维护：

- 当前游戏预算。
- 预算分配操作。
- 结算暂停开关。

说明：

- 预算用于确认结算时扣减。
- 分配预算会从公司侧增加该游戏可结算额度。
- 开启结算暂停后，该游戏不能确认新的结算批次。

### ECPM 同步

展示和维护：

- 自动同步开关，默认关闭。
- 自动同步频率，默认 3 小时。
- 频率预设：1、3、6、12、24 小时。
- 下次执行时间。
- 最近执行时间。
- 手动刷新范围。
- 手动刷新按钮。
- 最近同步任务列表。

说明：

- 自动同步开启后，系统按频率同步最近一个频率窗口的数据。
- 例如频率为 3 小时，则每次自动同步只处理最近 3 小时。
- 自动同步失败后不自动重试、不自动补拉历史。
- 管理员可以通过手动刷新补处理指定最近范围。

### 审计/任务历史

第一版可以复用现有审计日志和同步任务列表信息，按游戏聚合展示：

- 最近 ECPM 同步任务。
- 最近配置变更审计。
- 最近预算或结算相关操作。

如果现有审计接口不支持按游戏筛选，本阶段可先在配置页显示最近任务列表，并保留审计模块结构，后续扩展筛选能力。

## 后端数据模型

在 `Game` 增加自动 ECPM 同步配置字段：

- `ecpmAutoSyncEnabled Boolean @default(false)`
- `ecpmAutoSyncIntervalHours Int @default(3)`
- `ecpmAutoSyncNextRunAt DateTime?`
- `ecpmAutoSyncLastRunAt DateTime?`

字段语义：

- `ecpmAutoSyncEnabled`：该游戏是否开启自动同步。
- `ecpmAutoSyncIntervalHours`：同步频率，只允许 1、3、6、12、24。
- `ecpmAutoSyncNextRunAt`：下次自动同步应执行时间。
- `ecpmAutoSyncLastRunAt`：最近一次自动同步执行时间，无论成功或失败。

开启规则：

- 默认关闭。
- 当管理员从关闭切换为开启时，设置 `ecpmAutoSyncNextRunAt = now`，使系统立即执行一次。
- 当管理员修改频率时，如果自动同步已开启，重新计算下一次执行时间；若希望立即按新频率执行，管理员可使用手动刷新。

关闭规则：

- 当管理员关闭自动同步时，保留最近执行时间，清空或忽略 `ecpmAutoSyncNextRunAt`。
- 关闭后调度器不再执行该游戏。

在 `KuaishouEcpmSyncJob` 增加范围同步字段：

- `lookbackHours Int?`
- `startedDataHour String?`
- `endedDataHour String?`

字段语义：

- `lookbackHours`：本次任务覆盖的最近小时数。单点 `dataHour` 刷新可为空。
- `startedDataHour`：范围刷新内最早的小时点。
- `endedDataHour`：范围刷新内最新的小时点。
- `dataHour` 保留用于兼容现有单点刷新和列表展示。范围刷新时写入 `endedDataHour`，前端展示范围时优先使用 `startedDataHour`、`endedDataHour`、`lookbackHours`。

## 后端接口

### 游戏列表

现有管理员游戏列表接口返回新增字段：

- `ecpmAutoSyncEnabled`
- `ecpmAutoSyncIntervalHours`
- `ecpmAutoSyncNextRunAt`
- `ecpmAutoSyncLastRunAt`

前端游戏列表和配置页都从这里获取当前配置。

### 更新游戏配置

扩展现有游戏更新接口，或新增专用配置接口。推荐使用专用配置接口，让基础信息、预算结算、同步配置的验证更清晰。

建议接口：

`PATCH /api/admin/games/:gameId/config`

请求体按模块传递：

```json
{
  "name": "Runner",
  "gameSecret": "secret",
  "settlementPaused": false,
  "ecpmAutoSyncEnabled": true,
  "ecpmAutoSyncIntervalHours": 3
}
```

验证规则：

- `ecpmAutoSyncIntervalHours` 只能是 1、3、6、12、24。
- `game_app_id` 不允许更新。
- 所属公司不允许通过该接口迁移。
- 开启自动同步时，如果原状态为关闭，则设置 `nextRunAt = now`。

### 手动 ECPM 刷新

现有手动刷新接口扩展 `lookbackHours`：

`POST /api/admin/kuaishou/ecpm/refresh`

请求体：

```json
{
  "gameAppId": "game-1",
  "lookbackHours": 3
}
```

规则：

- `lookbackHours` 只能是 1、3、6、12、24。
- 如果传入 `openIds`，只同步指定 open_id；否则同步该游戏所有已知 open_id。
- `lookbackHours` 和旧的 `dataHour` 不建议同时传；如果为了兼容保留 `dataHour`，则 `dataHour` 表示单点刷新，`lookbackHours` 表示范围刷新。

## 同步范围

### 小时点拆分

快手 ECPM client 当前使用 `dataHour` 参数。为了支持“最近 N 小时”且不改快手接口模型，后端把范围拆成多个小时点逐个同步。

例如当前时间为 14:20，选择最近 3 小时，则生成 12:00、13:00、14:00 三个 `dataHour` 点。具体格式继续沿用现有 client 能解析的格式。

每次范围刷新创建一个同步任务，任务内循环多个小时点，最终汇总：

- 请求 open_id 数。
- 写入 rows 总数。
- source。
- 成功或失败状态。

### open_id 范围

自动同步默认同步该游戏下所有已知 open_id，包括已绑定和未绑定 open_id。

这样可以确保未绑定期间产生的收益也能先入库，后续用户绑定或查询时再关联。

## 自动同步调度

### 调度方式

采用后端进程内调度器。

API 服务启动后，调度器每分钟检查一次：

- `ecpmAutoSyncEnabled = true`
- `ecpmAutoSyncNextRunAt <= now`

的游戏。

本阶段不引入外部 cron、队列或分布式锁。这个选择适合当前单实例项目，后续多实例部署时再升级为外部调度或带锁任务队列。

### 到期执行

对每个到期游戏：

1. 检查该游戏是否已有 `RUNNING` 同步任务。
2. 如果已有运行中任务，跳过本轮，并把 `nextRunAt` 推到下一个周期。
3. 如果没有运行中任务，创建 `SYSTEM` actor 的同步任务。
4. 根据游戏频率生成最近 N 个小时点。
5. 对每个小时点调用 `KuaishouEcpmClient.refresh()`。
6. 保存所有返回 rows。
7. 完成任务，记录写入总数。
8. 更新 `ecpmAutoSyncLastRunAt = now`。
9. 更新 `ecpmAutoSyncNextRunAt = now + intervalHours`。

### 失败处理

如果自动同步失败：

- 当前任务标记 `FAILED`。
- 记录错误信息。
- 写入失败审计。
- 不自动重试。
- 不自动补拉历史。
- 不关闭该游戏自动同步。
- 更新 `ecpmAutoSyncLastRunAt = now`。
- 更新 `ecpmAutoSyncNextRunAt = now + intervalHours`。

管理员后续通过任务历史查看失败原因，并使用手动刷新选择回溯范围自行处理。

### 并发控制

同一游戏不允许并发自动同步。

如果某游戏上一轮任务仍为 `RUNNING`，本轮到期时跳过并推迟到下一个周期。这样避免重复调用快手接口、重复写入和资源抢占。

## 任务历史复用

继续复用 `KuaishouEcpmSyncJob`。

任务字段约定：

- `actorType = SYSTEM` 表示自动任务。
- `actorId = system` 表示系统调度器。
- `source` 仍为 `mock` 或 `kuaishou`。
- `dataHour` 对单点任务继续表示该小时点。
- `dataHour` 对范围任务写入最新小时点，保持旧列表和索引兼容。
- `lookbackHours`、`startedDataHour`、`endedDataHour` 用于精确展示范围和排查失败。

## 前端交互

### 游戏列表

游戏列表每行增加“配置”按钮。

点击后切换到游戏配置视图，并带入该游戏 ID。配置视图需要支持返回运营台游戏列表。

### 游戏配置视图

布局：

- 页面头部显示游戏名、`game_app_id`、所属公司、当前状态。
- 左侧模块导航。
- 右侧模块内容。

基础信息模块：

- 游戏名称输入。
- `game_app_id` 只读。
- `game_secret` 输入。
- 保存按钮。

预算与结算模块：

- 当前预算展示。
- 分配预算输入和提交按钮。
- 结算暂停开关。
- 配置说明。

ECPM 同步模块：

- 自动同步开关。
- 频率下拉选择，选项为 1、3、6、12、24 小时。
- 下次执行时间。
- 最近执行时间。
- 手动刷新范围选择，选项为 1、3、6、12、24 小时。
- 手动刷新按钮。
- 最近同步任务表。

审计/任务历史模块：

- 第一版展示该游戏最近同步任务。
- 如果审计筛选能力不足，保留审计入口和说明，后续补充按游戏筛选。

### 文案原则

配置说明采用后台操作说明，不使用营销语言。

示例：

- “开启后，系统会按所选频率同步最近一个频率窗口内的 ECPM 数据。”
- “失败不会自动重试。请在任务历史中查看错误，并使用手动刷新补处理。”
- “结算暂停开启后，该游戏不能确认新的结算批次。”

## 测试计划

### 后端测试

- `Game` 自动同步字段默认关闭，默认频率为 3。
- 保存游戏配置时只允许频率 1、3、6、12、24。
- 从关闭切换为开启时设置 `nextRunAt = now`。
- 关闭自动同步后调度器不执行该游戏。
- 调度器只执行到期游戏。
- 调度器跳过已有 `RUNNING` 任务的游戏。
- 自动同步生成最近 N 个小时点。
- 自动同步成功时完成任务并更新 `lastRunAt`、`nextRunAt`。
- 自动同步失败时标记任务失败，不重试、不补拉。
- 手动刷新 `lookbackHours` 拆成对应小时点。
- 手动刷新不传 `lookbackHours` 时保持兼容现有行为或使用默认单点行为。

### 前端测试

- 游戏列表渲染“配置”按钮。
- 点击配置后渲染游戏配置视图。

- ECPM 同步模块展示默认关闭、默认 3 小时。
- 保存自动同步配置请求参数正确。
- 手动刷新范围请求参数正确。
- 自动同步开启后显示“系统会立即执行一次同步”提示。

## 安全和资源边界

- 所有配置接口只允许管理员访问。
- 自动任务 actor 固定为 `system` / `SYSTEM`。
- 自动同步频率只允许预设值。
- 手动刷新范围只允许预设值。
- 同游戏自动任务不并发。
- 失败不自动补拉历史。
- 当前进程内调度器适合单实例；多实例部署前需要引入外部调度或分布式锁。
