# AI-KS 运营总览与基础体验完善设计

日期：2026-05-09

状态：已按当前讨论确认，等待最终审阅后进入实现计划。

## 背景

当前运营台已经覆盖公司、游戏、预算、结算、提现、快手授权与 ECPM 同步等功能，但信息主要分散在多个模块内，管理员进入页面后缺少一个稳定的“第一屏总览”来快速判断经营状态和风险。

同时，页面的加载、空态、错误态、禁用态和移动端表现目前存在局部差异，导致操作体验不一致，维护成本偏高。

## 目标

- 新增运营台默认入口“运营总览”视图，采用指标驱动、总览优先结构。
- 顶部固定四个核心指标：
  - 今日收益
  - 游戏预算余额
  - 待结算金额
  - 异常任务数
- 在同一页面提供近 7 日经营摘要、游戏排行、异常摘要。
- 在不做大规模重构的前提下统一基础体验：
  - loading
  - empty
  - error
  - success notice
  - 按钮禁用态
  - 移动端基础布局
- 第一版尽量复用现有 API 和前端状态，不以新增后端接口为前提。

## 非目标

- 不引入完整报表系统或数据仓库。
- 不在本阶段新增图表库作为前置依赖。
- 不重构全部历史页面或全部表格交互。
- 不自动触发所有游戏的结算预览计算。
- 不新增复杂路由体系或状态管理框架迁移。

## 设计原则

1. 先聚合、后扩展：优先利用现有数据完成业务价值，再补最小 API 缺口。
2. 口径可解释：每个指标必须能追溯数据来源与计算方式。
3. 组件边界清晰：`App.tsx` 管状态与请求，`OperationsWorkspace` 管展示，计算逻辑进入独立 helper。
4. 控制范围：第一版以“可用且一致”为目标，不追求一次性视觉或架构大改。

## 产品结构

在现有 `OperationsWorkspace` 内新增一个“运营总览”模块作为默认展示区，不单独拆新页面路由。运营台首屏信息组织如下：

1. 顶部指标卡区：4 个核心指标。
2. 中部经营摘要区：近 7 日收益/消耗等可得摘要数据。
3. 下部双列区：
   - 左侧：游戏排行。
   - 右侧：异常摘要。

若数据不足，模块使用统一空态文案，不显示误导性 `0`。

## 指标口径

### 1) 今日收益

- 目标语义：当前管理员“今天可见范围内”的收益。
- 数据来源：优先复用现有 ECPM/收益查询结果中可直接获得的当天数据。
- 第一版约束：若当前运营台没有稳定的全局今日收益源，则展示为“今日可见收益”并在文案层标注口径。

### 2) 游戏预算余额

- 目标语义：当前可见游戏的预算余额总和。
- 数据来源：`adminGames[].budget` 聚合。
- 作用范围：公司管理员仅统计其授权可见范围，超级管理员统计全部可见游戏。

### 3) 待结算金额

- 目标语义：当前上下文可确认的待结算金额。
- 数据来源优先级：
  1. 当前已加载的 `settlementPreview` 金额；
  2. 若无预览结果，则显示 `-` 并提示先执行结算预览。
- 设计约束：第一版不自动对所有游戏触发预览请求。

### 4) 异常任务数

- 目标语义：ECPM 同步任务中失败项数量。
- 数据来源：`kuaishouEcpmJobs` 与配置视图任务列表合并后去重。
- 统计规则：状态 `FAILED` 计为异常；按任务 ID 去重。

## 数据流与前端架构

新增独立聚合层 helper：

- 建议文件：`apps/web/src/lib/operationsOverview.ts`
- 核心函数：`buildOperationsOverview(input)`

输入建议：

- `adminGames`
- `adminCompanies`
- `settlementPreview`
- `settlementBatches`
- `kuaishouEcpmJobs`
- `configKuaishouEcpmJobs`
- （可选）当前日期时间，用于测试稳定性

输出建议：

- `metrics`: 指标卡数据结构
- `summary`: 近 7 日摘要数据
- `rankings`: 游戏排行数据
- `exceptions`: 异常摘要数据
- `emptyStateHints`: 空态提示信息

职责边界：

- `App.tsx` 继续负责请求、鉴权、全局错误处理和状态持有。
- `OperationsWorkspace.tsx` 只消费聚合结果进行渲染，不直接拼装业务口径。
- Helper 负责口径统一，保证页面逻辑可测试。

## UI 与交互

### 总览模块

- 使用现有 `MetricCard`、`Panel`、`StatusBadge`、`ReadoutGrid` 组件复用样式系统。
- 顶部指标卡保持同一高度与密度，避免随文案抖动布局。
- 摘要区与排行区以后台信息扫描效率为优先，不做营销式视觉。

### 统一基础体验（轻量全覆盖）

1. 空状态
   - 新增或扩展通用 `EmptyState` 表达。
   - 应用于无游戏、无排行、无异常、无可结算上下文。
   - 每个空态包含一条“下一步动作”提示。

2. 加载状态
   - 基于已有 `busyAction` 显示模块级 loading 文案。
   - 按钮继续沿用“进行中”文案，防止重复提交。

3. 错误与成功提示
   - 继续复用全局 `Alert`。
   - 统一总览页提示位置，避免被长内容挤出首屏。

4. 响应式
   - 指标卡：桌面 4 列，平板 2 列，移动端 1 列。
   - 摘要/排行/异常区：窄屏自动单列。
   - 历史表格保留现有横向滚动策略，不在本阶段深改。

## 风险与回退

风险：

- 指标口径受限于当前可得数据，可能出现“可见收益”与最终结算金额不一致。
- 旧模块对 `busyAction` 的占用可能影响总览加载文案一致性。

缓解：

- 在指标旁明确口径提示，避免误解为财务最终口径。
- 将口径与计算收敛在 helper，后续补后端汇总时最小改动替换。

回退：

- 若总览聚合引发不稳定，可保留新 helper 与测试，暂时关闭总览入口并恢复原模块顺序。

## 验收标准

- 管理员进入运营台首屏能看到总览模块。
- 四个核心指标按确认口径可计算并展示。
- 空态、加载态、错误态、禁用态在总览与相关模块表现一致。
- 移动端总览不出现主要内容重叠或不可读。
- 现有 Web 测试、lint、build 全部通过。
