# AI-KS 前端整体重构设计

日期：2026-05-08

状态：已按当前讨论确认，等待最终审阅后进入实现计划。

## 目标

按 `DESIGN.md` 一次性重构当前前端，把 `apps/web/src/App.tsx` 的单文件工作台拆成正式前端架构，并从登录页开始建立统一、干净、可扩展的后台体验。

本次重构只覆盖当前前端已有功能和展示方式，不新增后端未实现的业务能力。后端是唯一权限裁决点，前端只做展示组织和错误呈现。

## 设计方向

采用 `DESIGN.md` 的“温暖的业务后台极简风格”：

- 页面底色使用暖米色，避免纯白和冷灰。
- 内容面板使用白底、浅边框、少阴影。
- 珊瑚色只用于主按钮、当前导航和关键确认。
- 表格、筛选、状态、操作优先，不做营销页式 hero、渐变背景或装饰卡片堆叠。
- 控件圆角控制在 6-10px，页面间距紧凑但可读。
- 所有金额、数量、日期和状态以稳定、可扫描的后台样式展示。

## 架构

当前前端集中在一个 `App.tsx` 中，重构后拆成以下边界：

- `app/`：应用入口、路由装配、全局身份状态。
- `layouts/`：`AuthLayout` 和 `DashboardLayout`。
- `pages/`：登录页、游客收益查询页、用户工作台、管理员运营工作台。
- `components/ui/`：按钮、输入框、面板、指标卡、状态标签、表格、提示条等基础组件。
- `components/domain/`：收益表格、账号资料、提现批次、提现详情、审计日志等业务组件。
- `lib/api.ts`：集中 API 请求、响应解析和标准错误。
- `lib/auth.ts`：用户 token、管理员 token、游客模式状态。
- `lib/format.ts`：金额、日期、审计摘要等格式化。
- `types/`：API 响应和页面数据类型。

基础组件先在项目内实现，不引入 HeroUI。组件接口保持接近 shadcn/ui 的简单形态，后续如接入真正 shadcn/ui，可以逐步替换内部实现。

## 入口与身份展示

未登录时显示干净登录页。登录页只包含：

- 品牌标识和简短产品名称。
- 用户登录与管理员登录切换。
- 账号和密码输入。
- 登录按钮。
- 注册入口。
- “游客登录”按钮。

登录页不直接放收益查询表单。点击“游客登录”后进入完整后台外壳，但只展示“收益查询”一个菜单。游客查询页只开放单个 `open_id` 或可读 ID 的当天收益查询。

登录后的展示规则：

- 游客模式：完整后台外壳，只显示收益查询。
- 用户登录：显示收益查询和账号工作台相关入口。
- 管理员登录：显示收益查询、管理员运营、联调、提现审核和审计相关入口。
- 未登录：只显示登录页。

这些展示规则只是前端体验组织，不作为权限依据。

## 后端授权边界

前端不做复杂权限判断，不根据菜单隐藏来保证安全。所有权限以服务端返回为准。

在没有后端菜单或权限元数据的前提下，前端先按当前身份类型渲染入口。如果用户触发了无权接口，后端返回 `401` 或 `403`，前端只负责显示错误或清理失效登录状态。

后续如果后端提供菜单或权限配置接口，前端可以改为直接渲染后端返回的菜单和操作能力。

## 页面范围

### 登录页

登录页是独立页面，不带后台侧边栏。视觉保持干净、克制，不做营销化背景。

行为：

- 用户登录调用现有 `/accounts/login`。
- 管理员登录调用现有 `/admin/auth/login`。
- 注册入口调用现有 `/accounts/register` 或打开同页注册状态。
- 游客登录进入游客查询外壳，不请求登录接口。

### 游客收益查询

游客进入后使用 `DashboardLayout`，但导航只显示收益查询。

功能：

- 输入单个 `open_id` 或可读 ID。
- 调用现有 `/user/earnings?identity=...`。
- 展示当天收益概览：展示金额、原始金额、可读 ID、明细数量。
- 展示 ECPM 明细表。
- 无结果时使用温和空状态。

### 用户工作台

用户登录后可见。保留当前已有能力：

- 当前账号概览。
- 绑定 `open_id` 或可读 ID。
- 查询账号收益。
- 维护支付宝账号和真实姓名。
- 确认结算。
- 提交提现申请。
- 查看最近结算和提现批次摘要。

页面结构按 `DESIGN.md` 的顺序组织：页面标题、指标概览、操作面板、收益明细表。

### 管理员运营工作台

管理员登录后可见。保留当前已有能力：

- 管理员登录状态。
- 游戏端登录联调。
- 快手 ECPM 刷新。
- 提现审核列表。
- 提现详情。
- 打款成功、打款失败、关闭失败批次。
- 审计日志。

“联调工具”不作为游客或普通用户默认入口暴露，只在管理员运营区域内保留。

## API 与错误处理

API 调用集中到 `lib/api.ts`。当前仓库没有成体系的 API 文档，也没有 Swagger/OpenAPI 配置，因此前端先按现有 Nest controller 和当前前端调用事实整理本地 API 方法。

错误处理规则：

- `401`：清理当前身份 token，提示重新登录。
- `403`：保留当前页面，显示“无权限访问该操作”。
- `400` 和 `409`：展示后端返回的 `message`。
- 网络错误和 `5xx`：展示通用失败提示。
- 成功反馈使用低饱和绿色提示条。
- 错误反馈使用浅红底提示条。
- 按钮 loading 和 disabled 只用于防重复提交和输入不完整，不作为权限判断。

用户 token 和管理员 token 分开保存，避免互相覆盖。游客模式不写入登录 token。

## 组件规则

基础 UI 组件必须统一使用设计 token：

- `Button` 支持 primary、secondary、ghost、danger、compact。
- `InputField` 使用上方 label、下方 helper/error 的结构。
- `Panel` 负责白底、浅边框、标题和内容区。
- `MetricCard` 用于小型指标概览，不做装饰性大卡。
- `StatusBadge` 使用语义颜色：pending、success、danger、muted、info。
- `DataTable` 使用浅色表头、稳定列间距、数字列右对齐、移动端横向滚动。
- `Alert` 负责统一错误和成功提示。

页面不得重复手写按钮、输入框和表格样式。

## 响应式

桌面端使用左侧边栏、顶部上下文栏和主内容区。

移动端：

- 登录页保持单列。
- 后台侧边栏折叠为顶部或抽屉入口。
- 密集表格允许横向滚动。
- 操作面板从多列变成单列。
- 按钮高度不低于 40px。

## 测试与验证

本次重构的验证重点：

- 登录页能切换用户登录和管理员登录。
- “游客登录”进入完整后台外壳，且只展示收益查询入口。
- 游客可输入单个 `open_id` 或可读 ID 查询收益。
- 用户登录后能看到账号工作台入口。
- 管理员登录后能看到管理员运营入口。
- `401`、`403`、`400/409`、网络失败能按规则显示。
- 构建命令 `pnpm --filter web build` 通过。

如果测试框架在当前前端尚未建立，至少增加针对 API 错误解析和身份状态切换的轻量单元测试；页面行为通过构建和浏览器手动验证覆盖。

## 非目标

本次不做以下事项：

- 不新增后端权限系统或菜单元数据接口。
- 不把前端菜单隐藏当作安全边界。
- 不引入 HeroUI。
- 不实现后端未完成的代理、公司管理员、预算、配置中心等完整页面。
- 不重写旧项目代码，只从当前新前端和现有 API 出发。
