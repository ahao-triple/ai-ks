# ECPM 看板重构设计

更新日期：2026-05-11

## 1. 背景与目标

当前项目作为一个全栈结算系统经过多轮迭代，但实际业务诉求其实更聚焦：**项目最重要的是 ECPM 看板，需要功能全 + 操作简单**。现有页面（尤其是 `OperationsWorkspace` 那 10 个 tab 堆叠）对超级管理员和客户都偏程序化、看不懂。

本次重构目标：

1. **看板是所有角色登录后的唯一首页**——左侧一级导航按身份显示不同菜单，每个菜单下不再有 tab 嵌套
2. **以 ECPM 数据本身为主线**，把"金额/结算/入账/提现"等结算口径默认从看板隐藏，需要时再去单独菜单
3. **客户和测试者能看懂**：用大白话指标命名，去掉"raw_ecpms / config_snapshot / settlement batch"等内部术语
4. **支持实时验证 ECPM 链路**：用户详情页能逐条按时间倒序看到 ECPM 记录，配合自动刷新和静默限流，方便测试者拿测试机进游戏看广告后立即验证
5. **用户端同时做好移动端适配**

非目标（明确不做）：

- 不动后端结算/提现的核心逻辑
- 不引入工单系统（公司管理员看到异常仅展示，不带任何修复入口）
- 不引入新的支付通道
- 不做财务报表导出（属于 P2）

## 2. 整体设计原则

| 原则 | 说明 |
|---|---|
| **看板即首页** | 所有角色登录后默认打开"看板"菜单 |
| **左侧一级导航按身份显示** | 同一套布局，不同角色看到的菜单项数量、名称不同 |
| **菜单内不嵌套 tab** | 每个一级菜单下直接是列表 + 行内操作，不再有二级 tab |
| **看板默认不出现金额/结算/提现** | 顶部 KPI、记录列表的默认列只展示 ECPM 数据本身；金额相关字段通过"列设置"按需打开 |
| **公司、代理、平台分账比例不进用户端** | 普通用户看板严格不显示这些信息 |
| **下钻保留，操作弱化** | 数据视图（看板/列表/详情）允许下钻；管理操作集中在专用菜单内 |
| **限流"静默糊弄"** | 看 ECPM 数据这种纯查询，前端节流 + 后端兜底返缓存，不弹"操作过快"错误；敏感操作（提现、绑定）保持显式提示 |
| **术语保留 ECPM** | 直接用 ECPM 这个英文术语，不翻译成"千次曝光收益"等 |

## 3. 命名与口径

### 3.1 ID 体系

| 类型 | 格式 | 出现在 |
|---|---|---|
| 用户 ID（账号级身份） | `U-A8F3D2K`（U- 前缀 + 7 位） | 看板、列表、详情；用户能看到自己的 |
| 账号展示 ID（每个 open_id 对应） | `A8F3D2K`（纯 7 位，不带前缀） | 用户分组表的"账号"列、ECPM 单条记录的"账号"列 |
| 代理邀请码 | `L-A8F3D2`（L- 前缀） | 代理身份卡 |
| open_id 完整字符串 | 平台原值，如 `2_xxxxxxxxa8f3d2` | **只在超级管理员的"用户详情"页展开后展示**；其余位置一律不出现 |

> 现有 `apps/api/src/domain/identity` 已经在生成可读 ID。这里把"用户级"和"账号级"两层 ID 都纳入复用范围。具体前缀规则在实现阶段对照现状再定。

### 3.2 时间筛选

所有看板和下钻视图统一支持：今天 / 昨天 / 最近 7 天 / 最近 30 天 / 自定义。**默认"今天"**。

### 3.3 默认隐藏的字段

下列字段默认不出现，可通过"⚙ 列设置"打开：

- 展示金额（按平台展示金额比例折算后的金额）
- 入账状态（已入账 / 入账失败 / 待结算 等）
- 异常原因（预算不足、配置异常等）

## 4. 信息架构（IA）

### 4.1 各角色左侧菜单

| 角色 | 左侧菜单 |
|---|---|
| 超级管理员 | 看板 · 公司管理 · 游戏管理 · 提现管理 · 代理管理 · 结算管理 · 快手授权 · 分账与配置 · 权限 · 审计与维护 |
| 公司管理员 | 看板 · 游戏列表 · 用户查询 |
| 代理 | 看板 · 我的数据 · 我的提现 · 资料 |
| 普通用户 | 看板 · 我的提现 · 资料 |

> 每个一级菜单下不再嵌套 tab。点进去就是该模块的列表 + 行内操作按钮（操作仅对超级管理员展示）。

### 4.2 看板下钻栈（仅超级管理员、公司管理员）

```
全平台/公司   →   游戏   →   用户   →   用户详情
```

- **超级管理员**：根节点是"全平台"，可见所有公司、游戏、用户
- **公司管理员**：根节点固定为"自己的公司"，看不到其它公司
- 每一层都带：面包屑（可点击逐级返回）+ 数据更新时间 + 立即刷新按钮 + 时间筛选
- **代理不下钻**：代理只看到名下用户的聚合收益，看不到具体用户的 ECPM 记录、游戏明细、open_id

## 5. 看板详细设计

### 5.1 超级管理员看板

**顶部时间筛选 + 4 个核心 KPI**（纯 ECPM 口径）：

1. 今日 ECPM 条数（带"比昨天 ±X%"对比）
2. 今日平均 ECPM
3. 今日活跃游戏（"14 / 共 18"，副标"4 个无数据"）
4. 今日活跃用户

**主区上半（左 2/3 + 右 1/3）：**

- 左：最近 7 天趋势图（实线 = ECPM 条数 / 虚线 = 平均 ECPM）
- 右：异常提示区。包含：
  - 同步失败（红底）
  - 长时间无数据（黄底）
  - 不再展示"待审核提现 / 公司余额不足"等财务向告警

**主区下半：今日 ECPM 数据（按公司）表**

- 顶部切换："按公司 / 按游戏 / 按代理"三种分组维度
- 列：公司 / ECPM 条数 / 活跃游戏 / 活跃用户 / 平均 ECPM / 最高 ECPM
- 点击公司名进入下钻

#### 下钻视图

| 层级 | 列 |
|---|---|
| 公司列表（按今日 ECPM 条数降序） | 公司 / ECPM 条数 / 活跃游戏 / 活跃用户 / 平均 ECPM / 最高 ECPM |
| 游戏列表（按今日 ECPM 条数降序） | 游戏 / ECPM 条数 / 活跃用户 / 平均 ECPM / 预算余额 |
| 用户列表（按今日 ECPM 条数降序） | 用户 ID（U-XXXXXXX） / 所属代理 / 今日条数 / 最近 ECPM / 最近活跃 |
| 用户详情 | 见 §6 |

> 列表标题统一用"X 列表（按 Y 降序）"的中性语义，不再使用"排行 / TOP / 排名"等竞技性字眼。

### 5.2 公司管理员看板（纯只读）

**核心定位：超级管理员看板的受限只读版**。所有"修复异常 / 提交工单"等动作入口去掉。

**顶部：欢迎 + 公司切换器**（多家授权公司时下拉选，最后一项"全部授权公司合并"）+ 时间筛选

**4 个 KPI**（公司范围）：今日 ECPM 条数 / 今日平均 ECPM / 今日活跃游戏 / 今日活跃用户

**主区：**

- 全宽趋势图（最近 7 天，无异常区）
- 游戏数据列表：游戏 / ECPM 条数 / 活跃用户 / 平均 ECPM / 最高 ECPM / 状态（同步状态作为只读列展示）

下钻保留（点游戏名 → 用户列表 → 用户详情），看不到 open_id 完整值。

### 5.3 代理看板（极简两块）

**第一块：代理身份卡（顶部金色调高亮）**

- 邀请码 `L-A8F3D2` + 一键复制 + 推广链接
- 三个数字：被代理用户（41 人）/ 名下今日总收益（¥ 124.80）/ 我的分账今日（¥ 18.72）

**第二块：名下用户收益列表**

- 列：用户 ID / 今日收益 / 今日 ECPM 条数 / 累计收益 / 注册时间 / 最近活跃
- **不可下钻**：用户 ID 不带链接，代理无权查看名下用户的 ECPM 单条记录、游戏明细、open_id
- 顶部搜索框（按用户 ID）+ 底部翻页
- 不活跃用户灰色降透明度

**代理工作台不展示代理自己的 ECPM 数据**——这部分搬到二级菜单"我的数据"，内容与普通用户看板完全一致。

### 5.4 普通用户看板

业务模型：**1 个用户账号 → N 个游戏 → 每个游戏 M 个手机/账号**。每个手机/账号对应一个 open_id，但 open_id 一字不显示，仅展示 7 位"账号展示 ID"。

**顶部欢迎条**：`你好 U-A8F3D2K · 3 个游戏 · 6 个账号 · 最近活跃 3 分钟前`

**4 个 KPI**：

1. 今日 ECPM 条数
2. 今日平均 ECPM
3. 游戏数（"3 · 2 个今日活跃"）
4. 账号总数（"6 · 5 个今日活跃"）

**主区上半："我的游戏与账号"分组表**

- 游戏汇总行（深色背景）：游戏名 + 账号数 + 今日条数 / 平均 / 累计 / 最近活跃（合计）
- 缩进的账号行：展示 ID / 今日条数 / 平均 ECPM / 累计 / 最近活跃 / 活跃状态徽章
- "+ 绑定新游戏 / 新账号"按钮在表头右上

**主区下半：我的 ECPM 记录**

- 与超级管理员下钻到用户层共用同一组件（见 §6）
- 列默认：今日序号 / 时间（精确到秒）/ ECPM / 游戏 / 账号
- 顶部筛选：游戏（全部 / 各游戏） + 账号（全部 / 各账号）
- 自动刷新（默认每 10 秒）+ 立即刷新

**严格规则：**

- 不显示公司、代理（即使用户绑定了代理）、平台分账比例
- 不显示"立即提现 / 可提现余额"等结算/提现导流
- ECPM 单条记录默认只 5 列；展示金额、入账状态、异常原因通过"列设置"按需打开

## 6. ECPM 单条记录组件（通用）

这是整个系统最关键的视图，用于实时验证"看广告 → ECPM 入库 → 收益入账"链路。

**字段：**

| 列名 | 默认 | 说明 |
|---|---|---|
| 今日序号 | ✅ | "第 N 条 / 第 1 条 🎯"标识；按用户当天累计算 |
| 时间（精确到秒） | ✅ | `2026-05-11 14:34:18` |
| ECPM | ✅ | 原始 ECPM 数值 |
| 游戏 | ✅ | 游戏名 |
| 账号 | ✅ | 7 位展示 ID |
| 数据来源 | ✅（部分视图） | 快手同步 / 手动刷新 / 自动同步 |
| 展示金额 | ⚪ | 默认隐藏，列设置开启 |
| 入账状态 | ⚪ | 默认隐藏，列设置开启 |
| 异常原因 | ⚪ | 默认隐藏，列设置开启 |

**交互：**

- 顶部状态条："上次刷新 X 秒前 · 下次自动刷新倒计时 N 秒 · 本次新增 K 条"
- 自动刷新开关 + 间隔（5/10/30 秒），可由用户控制
- 立即刷新按钮（受限流约束，下文详述）
- 新进入的记录顶部黄底高亮 2 秒淡出
- 失败/异常记录默认不强调（避免给人"出问题"错觉）；通过"⚙ 列设置"打开"异常原因"列才看得到

**测试者使用流程**：
1. 打开测试账号详情页（"今日 0 条"）
2. 进游戏看一个广告
3. 点 ⟳ 立即刷新 → "第 1 条 🎯" 黄底浮现
4. 再看 → 第 2 条、第 3 条逐条出现
5. 也可开自动刷新放着不管

## 7. 限流与"静默糊弄"策略

### 7.1 适用范围

| 场景 | 策略 |
|---|---|
| 看 ECPM 数据 / 看板刷新 / 列表筛选 | **静默节流**：前端节流 + 后端缓存兜底，不弹错误 |
| 提现申请 / 绑定新游戏 / 修改资料 | **显式提示**：超出限频弹"已提交太多次，请 X 分钟后再试" |

### 7.2 静默节流的具体行为

**用户疯狂点 "⟳ 立即刷新"：**

| 用户行为 | 前端 | 后端 | 用户感知 |
|---|---|---|---|
| 第 1 次点 | 立即发请求；按钮短暂禁用 1 秒 | 正常处理 | 看到新数据 |
| 5 秒内连续点 10 次 | 防抖 1 秒 + 节流 5 秒窗口；多余点击丢弃，按钮不显式禁用 | 只收到 1 次请求 | 每次点都见旋转动画 + "✓ 已是最新"，不出错误 |
| 5 秒后再点 | 节流窗口已过 → 真正发请求 | 正常返回 | 看到新数据 |
| 前端节流被绕过 | — | 按用户 ID 限流，超出时**返回上次缓存数据 + 200**，不返回 429 | 看到旧数据但无差别 |

**切换游戏 / 账号筛选：** 300ms 防抖 + 取消上次未完成请求；服务端永远只处理最后一次。

**自动刷新 + 手动刷新交叉触发：** 共享同一节流窗口；手动刷新会重置自动倒计时。

### 7.3 各操作的限流参数（建议默认）

| 操作 | 前端节流 | 后端限流 | 兜底 |
|---|---|---|---|
| ⟳ 立即刷新看板/记录 | 5 秒窗口 | 12 次/分钟 | 返回缓存 |
| 自动刷新 | 最快 5 秒/次 | 同上 | 同上 |
| 游戏 / 账号筛选切换 | 300ms 防抖 | 不限 | 取消上次请求 |
| + 绑定新游戏 | 提交后按钮禁用 | 5 次/小时 | **显式提示** |
| 提现申请 | 提交后按钮禁用 | 3 次/天 | **显式提示** |

## 8. 移动端适配（重点：用户端）

普通用户多在手机上使用，必须支持手机宽度（默认按 360px 设计）。

**布局变更（对比桌面）：**

| 区域 | 桌面 | 移动 |
|---|---|---|
| 顶部 | 4 个时间筛选按钮平铺 | sticky 栏 + 时间筛选下拉 |
| 4 KPI | 1 行 4 列 | 2 行 2 列 |
| 我的游戏与账号 | 整张表，游戏汇总行 + 账号行 | 折叠手风琴卡，每个游戏一张大卡，点击展开账号 |
| ECPM 记录 | 完整表格 | 精简卡片堆叠（每条记录一张小卡） |
| 刷新控件 | 表头右上 | **底部固定栏**：自动刷新开关 + ⟳ 立即刷新大按钮 |
| 列设置 | 弹出小面板 | 全屏底部抽屉 |

**手机端默认值差异：**

- **自动刷新默认关闭**（耗电、流量）
- 单条记录列默认更精简（仅"序号 / 时间 / ECPM / 游戏·账号"）

## 9. 提示与状态设计

| 状态 | 设计 |
|---|---|
| 加载 | 骨架屏（不要纯转圈） |
| 空状态 | 友好引导（如"还没有 ECPM 记录，绑定一个游戏并看广告后回来这里点刷新"） |
| 成功 | 顶部 toast，绿底 |
| 静默节流 | 蓝底 toast `✓ 数据已是最新`（不出错误） |
| 显式错误（仅敏感操作） | 弹窗 / 红底 toast，明确原因 + 建议 |
| 引导（首次使用） | 黄底说明卡，3 步流程 |

## 10. 后端配合需要

### 10.1 数据范围 API

每个角色的看板数据需要按角色权限过滤：

- 超级管理员：全平台
- 公司管理员：公司 ID 列表（来自 `CompanyAdminScope`）
- 代理：名下用户 ID 列表
- 普通用户：自己的用户 ID

> 现有 `features/business-closure / settlement-admin / agent / account` 等模块已经具备按身份过滤的查询能力，本次主要是补充"看板维度聚合"的查询接口。

### 10.2 ECPM 单条记录 API

- 支持秒级时间字段查询
- 支持按 user_id / game_id / 账号展示 ID 三层筛选
- 返回字段精简成看板组件需要的列（序号 / 时间 / ECPM / 游戏 / 账号 / 数据来源）
- 支持"今日序号"语义（按用户当天累计编号）

### 10.3 限流后端兜底

- 按用户 ID + 接口路径维度限流
- 超出时返回 200 + 上次缓存数据 + `Cache-Hit: true` header
- 前端可以无差别处理

### 10.4 公司管理员"我的公司 / 我的游戏"已存在

直接复用现有 API，不需要新增。

## 11. 不在本次范围

- 真实快手授权 / token 维护工作流（保持现状）
- 真实支付宝打款（保持模拟）
- 财务报表 / 导出
- 工单系统
- 代理推广素材（落地页 / 二维码）—— 仍是后续 P1
- 公司管理员的写权限（保持只读 MVP）

## 13. 验收口径

视觉/交互验收（人工）：

- 4 个角色登录都能进入对应看板
- 看板首屏不出现"金额/结算/入账/提现"字样（除代理身份卡的 2 个金额数字）
