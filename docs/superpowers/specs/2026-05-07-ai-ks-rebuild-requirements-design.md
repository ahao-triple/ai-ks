# AI-KS 重构需求设计

日期：2026-05-07

状态：已按当前讨论确认，等待最终审阅后进入实现计划。

## 设计原则

本次重构只从旧代码提取业务需求和外部接口事实，不继承旧代码的实现方式、模块组织、命名习惯或前端写法。新系统采用“配置中心 + 账务快照驱动”的方案：原始数据不丢，金额和权限可配置，历史记录保存当时的配置快照，提现打款按明细独立执行。

前端合并为一个应用，不再使用 HeroUI。组件体系按已确认的设计规范采用 `shadcn/ui + Tailwind CSS + Radix UI + lucide-react + TanStack Table`。

## 系统边界与身份模型

系统由一个统一前端和一个后端服务组成。登录后按身份、权限和数据范围显示不同菜单。

核心身份：

- 用户：查询收益、注册账号、绑定多个游戏 `open_id`、维护支付宝、发起提现、查看打款状态。
- 代理：查看收益和名下用户，维护自己的支付宝收款信息，不查看用户提现账号明细，不审核，不主动发起提现。
- 公司管理员：只访问超级管理员分配的公司和游戏范围；默认几乎无权限，需要逐项授权。
- 超级管理员：最高权限，管理公司、游戏、代理、权限、平台 token、金额配置、预算、结算、提现审核和打款异常。

核心归属关系：

- 公司拥有游戏，公司之间数据隔离。
- 游戏有自己的 `game_app_id` 和 `game_secret`。
- 平台 MAPI token 是全局配置，不归属公司。
- `open_id` 全局唯一，属于用户。
- 用户属于代理，用户可以立即换绑代理。
- 代理是平台级角色，高于公司，可推广所有游戏。

## 平台接口与收益同步

游戏侧登录时，游戏把 `js_code + game_app_id` 发给后台。后台根据后台创建游戏时配置的 `game_app_id/game_secret` 调快手小游戏登录接口换取 `open_id`。系统为 `open_id` 生成一个最多 7 位的可读 ID，用于用户查询和绑定。

平台侧做全局配置：

- 全局维护 MAPI `app_id`、MAPI `secret`、`advertiser_id`、`access_token`、`refresh_token` 和 token 状态。
- 获取 token 支持回调自动换 token，也支持超级管理员手动粘贴 `auth_code` 兜底。
- token 自动刷新；刷新失败时同步任务暂停并标记异常，超级管理员可重新授权。
- 公司不维护平台 token，公司与快手平台授权无耦合。

收益同步使用快手 ECPM 报表接口。`cost` 按接口单位“厘”作为整数保存，金额展示时再转换为元。

同步方式：

- 定时同步：按游戏拉取近期数据，作为主流程。
- 用户查询触发：用户用 `open_id` 或可读 ID 查询时，对单个用户做轻量刷新。
- 超级管理员手动刷新：按游戏维度全量刷新，覆盖系统里还没有出现过的 `open_id`。

同步回来的数据先保存为原始收益记录。如果 `open_id` 已绑定用户，就关联到用户；如果还未绑定，也先保存，等用户查询或绑定后再关联。重复 ECPM 数据不能重复入账。

## 金额展示、结算与预算

系统采用“原始收益 -> 展示金额 -> 待结算 -> 可提现”的链路。快手原始收益不直接变成用户余额。

金额分三层：

- `raw_cost`：快手原始金额，用于来源记录和后台核对。
- `display_amount`：用户侧、游客查询和默认结算基础使用的展示金额。
- `settlement_amount`：确认结算后在 `display_amount` 基础上生成的最终可提现、代理收益、手续费和预算扣减金额。

用户看到的金额沿用旧后台的业务思路：不直接展示快手原始 `cost`，而是通过超级管理员配置的“金额展示方案”生成展示金额。新系统只沿用这个思路，不沿用旧实现。

金额展示方案支持：

- 按比例缩放，默认 `50%`。
- 按游戏覆盖比例。
- 最大展示金额封顶。
- 最小金额、中位金额、顶包金额规则。
- 按概率过滤某些记录为 `0`。
- 大额收益按概率真实展示、按比例展示或封顶展示。
- 是否启用模型方案。
- 方案作用范围支持全局、公司、游戏、代理或组合覆盖。

结算配置：

- 用户基础比例默认 `50%`，由超级管理员配置。默认情况下，这个比例用于金额展示方案生成 `display_amount`，不会再对 `display_amount` 二次乘 `50%`；如果超级管理员启用独立结算比例，则按独立配置生成结算快照。
- 代理点数由超级管理员配置，支持全局默认、按代理、按游戏、按代理+游戏覆盖。
- 手续费比例默认 `3%`，由超级管理员配置。
- 手续费收款支付宝账户由超级管理员配置。
- 手续费承担方式可配置，默认从收款方应得金额中扣，也可配置为从公司或游戏预算额外出。
- 最低提现金额、单笔打款上限、余额预警线、同步频率、补结算范围等数值规则都进入配置中心。

结算权限：

- 收益同步后先进入待结算。
- 默认只有超级管理员能确认结算。
- 超级管理员可以把指定公司和游戏范围的结算确认权分配给公司管理员。
- 公司管理员只能处理授权范围。
- 每次结算保存配置快照、操作人、时间、金额、公司、游戏、用户和代理链路。

预算逻辑：

- 公司有余额，游戏有预算。
- 超级管理员手动给公司充值或调整余额。
- 超级管理员把公司余额分配到游戏预算。
- 游戏结算消耗游戏预算。
- 某个游戏预算不足时，只暂停该游戏的新结算，不影响其他游戏。
- 预算不足期间，原始数据仍继续同步和保存。
- 预算恢复后，是否对暂停期间补结算，由超级管理员手动选择游戏和时间范围处理，流程保持简单。

配置变更只影响之后新生成的数据；已生成的收益、结算、提现、打款明细都保留当时快照。

## 用户、代理与绑定关系

游客查询：

- 游客可用 `open_id` 或 7 位内可读 ID 查询收益。
- 默认查询当天 `00:00:00` 到 `23:59:59`。
- 用户侧只展示游戏信息和金额，不展示公司信息。
- 游客不能提现。
- 必须注册账号、绑定 `open_id`、维护支付宝信息后才能发起提现。

用户账号：

- 用户用账号密码注册和登录。
- 不做手机号验证码。
- 用户端不提供默认修改密码入口。
- 密码由超级管理员重置。
- 用户自己维护支付宝账号和姓名，修改后立即生效。
- 新提现明细使用创建时的支付宝信息快照，旧单不受后续修改影响。

绑定规则：

- 一个用户可以绑定多个游戏的多个 `open_id`。
- `open_id` 一旦绑定用户，不允许用户自己解绑或转移。
- `open_id` 先有收益、后被用户绑定时，未结算收益可关联到该用户；已结算历史不自动改。
- 用户可以立即换绑代理。
- 换绑代理后，下一笔新结算或新提现批次按新代理走，历史账和已生成打款明细不改。
- 系统记录代理换绑历史：原代理、新代理、操作时间和来源。

代理规则：

- 代理账号只能由超级管理员创建和管理。
- 代理可以推广所有游戏。
- 代理支持两级：用户直属代理和上级代理。
- 用户没有代理时，代理收益归到底层默认代理。
- 底层默认代理支持全局默认和按游戏覆盖。
- 用户可以通过代理注册链接注册，也可以注册后输入代理邀请码绑定。
- 代理自己维护支付宝信息，修改后立即生效。
- 代理后台只看收益和名下用户列表，不看具体 `open_id`、用户提现、支付宝信息，也不主动发起提现。

## 提现、代理打款、手续费与异常处理

提现必须由用户发起。用户可以按单个游戏提现，也可以多个游戏一起提现。前端可以展示一个提现批次，后端按游戏、公司、`open_id`、代理和手续费拆成多条打款明细。

提现批次包含：

- 用户本人打款明细。
- 直属代理打款明细。
- 上级代理打款明细。
- 默认底层代理打款明细，如果用户没有代理。
- 手续费打款明细，收款方为超级管理员配置的支付宝账户。

审核规则：

- 用户发起提现后，相关金额冻结。
- 支付宝打款失败后保持冻结，不自动退回。
- 默认超级管理员审核。
- 超级管理员可以把指定公司和游戏范围的审核权分配给公司管理员。
- 审核时看到整组“用户 + 代理 + 手续费”打款明细。
- 审核可以整批确认，但执行结果按明细独立落状态。

打款规则：

- 按支付宝自动打款实现。
- 用户、代理、手续费三类明细互不阻塞。
- 用户成功不等代理成功，代理成功不等用户成功，手续费成功或失败也不影响两方。
- 每条明细保存收款账号快照、金额、手续费规则快照、配置快照、支付宝请求摘要和响应摘要。
- 支付宝失败时记录错误码和错误信息。
- 可重试错误由管理员手动重试。
- 不可重试错误进入人工处理。
- 管理员可以重试、标记人工处理、关闭并退回冻结金额。

批次状态由明细汇总得出，例如待审核、待打款、部分成功、全部成功、部分失败、已关闭。

## 权限、配置中心与前端视图

权限默认收紧，尤其是公司管理员。超级管理员创建公司管理员后，需要显式分配数据范围和操作权限。

权限分两层：

- 数据范围：可访问哪些公司、哪些游戏。
- 操作权限：可查看数据、确认结算、审核提现、重试打款、管理用户、管理游戏、查看预算、导出数据、查看代理信息、查看代理金额等。

公司管理员默认不能看到代理信息和代理收益。超级管理员可以单独配置“可见代理身份”和“可见代理金额”。公司管理员不能创建其他公司管理员，也不能越权访问其他公司或游戏。

配置中心由超级管理员维护，包含：

- 平台 token 与授权配置。
- 游戏登录凭证。
- 用户金额展示方案。
- 用户分成比例。
- 代理点数规则。
- 手续费比例、承担方式、收款支付宝账户。
- 公司余额、游戏预算、预警线。
- 结算确认权限和提现审核权限。
- 同步频率、按游戏刷新、补结算策略。
- 默认底层代理及游戏覆盖。
- 支付宝打款参数，如单笔上限、最低金额、错误重试策略。

前端工作台：

- 用户工作台：查询、绑定、收益、提现、支付宝。
- 代理工作台：收益概览、名下用户、收款信息。
- 公司管理员工作台：授权范围内的数据、结算、提现审核、游戏视图。
- 超级管理员工作台：全局运营、配置、权限、平台接口、预算、异常处理。

## 数据流、异常与审计

核心数据流：

1. 游戏登录：游戏传 `js_code + game_app_id`，后台换 `open_id`，生成可读 ID。
2. 收益同步：快手 ECPM 原始数据入库，保留 `raw_cost`，按展示金额方案生成 `display_amount`。
3. 结算确认：待结算收益按配置快照生成用户收益、代理收益、手续费和游戏预算扣减。
4. 提现申请：用户发起提现，余额冻结，生成批次和多条打款明细。
5. 审核打款：管理员审核后逐条支付宝打款，明细独立成功或失败。
6. 配置变更：只影响之后新生成的数据，历史数据保留当时快照。

异常处理：

- 平台 token 失效：同步暂停，标记异常，超级管理员重新授权。
- 快手同步失败：记录任务失败原因，可重试，不重复入账。
- 游戏预算不足：暂停该游戏新结算，原始数据继续保存。
- 金额展示方案异常：相关批次进入异常状态，不能自动结算，等待超级管理员处理。
- 支付宝打款失败：单条明细保持冻结，记录错误码，可重试或人工处理。
- 权限越界：直接拒绝，并记录审计日志。

审计日志覆盖：

- 配置变更。
- 代理换绑。
- 用户和代理支付宝信息修改。
- 结算确认。
- 提现审核。
- 打款重试、人工处理、关闭退回。
- 平台 token 授权和刷新。

测试重点：

- 金额展示方案和配置优先级。
- 原始金额、展示金额、结算金额的快照一致性。
- 游戏预算不足暂停和补结算。
- 提现冻结、失败保持冻结、关闭退回。
- 用户、代理、手续费明细独立成功或失败。
- 公司管理员数据范围和操作权限隔离。
- 代理信息默认不可见和按权限开放。
- 平台 token 刷新和重新授权。
- ECPM 重复数据去重。

## 外部资料与旧代码参考边界

外部接口事实来自快手官方 ECPM 报表文档和【快小游】Marketing API 接入实时 ECPM 操作指引。旧代码仅用于理解历史业务需求，尤其是 `cost_client` 这类用户展示金额处理思路；新系统不得继承旧代码的实现结构和写法。
